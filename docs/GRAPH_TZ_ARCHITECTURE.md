# Graph/TZ UI Architecture

This sidecar treats the graph as the practical source of truth for the TZ. The
owner should be able to filter to a small branch, inspect one node at a time,
and use visual keys instead of long specification documents.

## Safety Boundary

- Node.js only for this sidecar.
- Existing PHP viewer remains quarantined reference material.
- No submit/upload/send flows are called.
- No `.env`, cookies, saved browser profiles, private CVs, or private runtime
  data are read.

## Superpower Flow Integration

No canonical local implementation named `superpower` was found in the safe
search scope. The current integration point is:

```text
GET /api/superpower-flow
GET /api/graph-model/job-apply-full
```

Assumed flow:

1. `focus`: owner or agent narrows the graph to a small branch.
2. `attach_agent_branch`: each task node can attach an agent-branch subtree.
3. `produce_artifact`: branch work emits decisions, code refs, tests, and visual
   artifacts.
4. `visual_key_update`: latest release/test preview updates the node visual key.
5. `human_review`: owner inspects the branch graphically.
6. `release_or_replace_agent`: the branch is released, or the CLI agent is
   replaced while preserving branch identity and artifacts.

An agent branch is an overlay above one CLI agent type, for example Codex or
Claude. Replacement is allowed when the human sees context overflow, degraded
reasoning, stale state, or explicitly requests replacement. Replacement must
preserve node id, branch id, artifacts, decisions, open blockers, and handoff.

## Variant A

`/babylon-3d.html`

Babylon.js 3D force graph:

- spheres/boxes by task type;
- group/status colors;
- spring attraction along edges;
- node repulsion and center gravity;
- hover/click details;
- pause/reset/search/group controls.

## Modern 2D Cytoscape Direction

`/cytoscape-2d.html`

This replaces the old PHP/web prototype direction with current Cytoscape.js:

- modern Cytoscape.js CDN prototype;
- no PHP upload/proxy endpoints;
- filters by group/status/search;
- layouts: cose, breadthfirst, concentric, circle, grid;
- node focus and linked-neighborhood mode;
- export links for Markmap, OPML, and FreeMind.

Future import/export work should operate on graph JSON through Node routes and
explicit local files, not PHP endpoints.

## Variant B

`/flow-variant-b.html`

Dependency-free ReactFlow-style prototype for A/B testing:

- dense cards grouped into lanes;
- source/target handle visual language;
- SVG edge routing;
- visual-key placeholder per node;
- agent branch metadata in each card;
- search dimming instead of forcing full reading.

Production variant B can move to `@xyflow/react` once dependencies are approved.
The React Flow concepts to preserve are nodes with arbitrary data, edges with
source/target handles, controls, minimap, custom nodes, and custom edges.

## Mind-Map Exports

Runtime endpoints:

```text
/api/exports/job-apply-full/markmap
/api/exports/job-apply-full/opml
/api/exports/job-apply-full/freemind
```

Local file generation:

```powershell
npm.cmd run export:mindmaps
```

Formats preserve hierarchy, label, status, owner, group, detail, visual key, and
agent branch metadata where the format can carry it.

## Visual Keys

Current visual keys are placeholders:

```json
{
  "kind": "placeholder",
  "label": "Latest active visual key",
  "artifactRef": null
}
```

The next integration should update `visualKey` from safe release/test artifacts,
for example screenshots generated by UI smoke tests. Private browser material and
CVs must stay outside this sidecar.
