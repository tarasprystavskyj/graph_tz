# Як я перетворюю довге ТЗ на інтерактивний граф для роботи з агентами

> Чернетка від `article-agent` для DOU. Текст навмисно написаний як draft:
> перед публікацією варто додати скріншоти, короткий GIF і прибрати внутрішні
> назви, які не потрібні зовнішньому читачу.

## Коротко

Я роблю невеликий sidecar-проєкт для візуалізації технічного завдання як графа.
Ідея проста: замість того, щоб читати довгий текстовий документ, людина бачить
вузли, ребра, статуси, візуальні ключі, піддерева агентів і test-flow для кожної
фічі.

Це не task tracker у звичному сенсі. Це спроба зробити граф практичним source of
truth для розробки: що треба зробити, чому воно пов'язане з іншими частинами,
як це тестувати, хто або який агент цим займається, і який артефакт підтверджує
результат.

## Проблема

Довгі ТЗ мають дві типові проблеми:

- їх важко читати, особливо коли контекст швидко росте;
- вони погано показують зв'язки між рішеннями, тестами, UI-артефактами й
  кодовими змінами.

Коли в роботу додаються AI-агенти, проблема посилюється. Агент може виконати
окрему задачу, але людині все одно треба швидко зрозуміти:

- де ця задача в загальному контексті;
- що змінилось;
- як перевірити результат;
- чи не треба замінити агента через втрату контексту або деградацію reasoning.

## Рішення

Я виніс UI в окремий Node.js sidecar `graph-ui`. Він не торкається основних
production-flow, не читає секрети й не має доступу до чутливих даних. Його
задача - тільки показувати граф і зберігати локальні graph edits.

Основні режими:

- Babylon.js 3D graph: force-directed сцена з spring attraction, repulsion,
  damping і focus animation.
- Cytoscape.js 2D graph: щільніший режим для фільтрів, layout-ів і швидкого
  огляду.
- Flow-style variant: карткове представлення, схоже на React Flow, для
  порівняння читабельності.
- Mind-map exports: Markmap, OPML, FreeMind.

## Що є у вузлі

Кожна нода має:

- коротку назву;
- detail;
- group/status/workflow state;
- owner або агент;
- keyword icons;
- visual key;
- optional screenshot/image preview;
- testFlow: як саме перевіряти фічу;
- testComments: коментарі тестування від людини або агента.

Нові задачі в графі блимають червоним контуром, доки власник не клікне по них.
Це простий механізм "я бачив, що задача додалась".

## Agent-flow

Проєкт має два логічні агенти:

1. `browser-qa-agent`
   - відкриває сторінку в браузері через Chrome DevTools Protocol;
   - проходить ключові UI-функції;
   - для кожної ноди додає `testFlow`;
   - пише `testComments`.

2. `fixer-agent`
   - бере fail-коментарі від browser QA;
   - виправляє код;
   - після цього цикл QA повторюється.

Це поки не повністю автономна система агентів, але вже є відтворюваний script:

```bash
npm run agent-cycle
```

Він проходить поточний граф, тестує core UI й записує test-flow metadata назад у
graph edits.

## Чому не просто Jira/Linear/Markdown

Markdown зручний для читання зверху вниз. Але він слабко показує:

- паралельні залежності;
- гілки агентів;
- візуальні ключі;
- статуси на різних рівнях;
- шлях від задачі до тестового артефакту.

Jira/Linear краще трекають задачі, але часто перетворюють систему на список.
Мені потрібен не список, а карта зв'язків.

Граф дає інший режим мислення: можна зменшити когнітивне навантаження фільтрами,
клікнути одну ноду, подивитись її піддерево, згенерувати текстовий опис або
таблицю, а потім повернутися назад у граф.

## Технічна архітектура

Sidecar мінімальний:

- Node.js static/API server без npm-залежностей;
- Babylon.js і Cytoscape.js через CDN для прототипу;
- graph JSON у Cytoscape-compatible форматі;
- `data/graph-edits/*.json` для локальних overlay edits;
- browser smoke tests через Chrome DevTools Protocol;
- Apache reverse proxy на VPS.

Ключовий принцип: graph-ui не має торкатися safety-critical flow основного
проєкту. Це окремий шар візуалізації й навчання.

## Що вже працює

- 3D force graph.
- 2D Cytoscape layout modes.
- Context menu по right-click.
- Context search: агент підсвічує ноди, які схожі на відповідь на питання.
- Details panel по кліку на ноду або label.
- Редагування title/detail.
- Paste/upload image preview.
- Test comments.
- `testFlow` для нод.
- Export у mind-map формати.
- Deploy на VPS.

## Що ще сире

- Потрібні нормальні скріншоти/візуальні артефакти для кожної гілки, а не тільки
  symbolic placeholders.
- Потрібен кращий relevance scoring для context search.
- Потрібне зручніше редагування самого графа, а не тільки node details.
- Потрібно більше browser QA сценаріїв.
- Потрібен lifecycle для agent replacement, коли агент втрачає контекст.

## Висновок

Граф не замінює повністю текст, таблиці чи task tracker. Але він може стати
центром, з якого генеруються всі ці представлення.

Для мене головна цінність у тому, що граф допомагає не читати весь документ
щоразу. Він дає можливість сфокусуватись на потрібній частині, побачити зв'язки
і одразу зрозуміти, як перевіряти конкретну фічу.

Це поки експеримент, але вже видно напрям: технічне завдання може бути не тільки
документом, а інтерактивною картою роботи людини й агентів.
